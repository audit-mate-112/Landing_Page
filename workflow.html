Audit Lifecycle & Workflow

A technical walkthrough of how AuditMate processes compliance evidence.

01. Trigger & Validation

When a scan is initiated, the API performs a three-tier validation check before spawning background processes.

POST /api/v1/audits/start/
> Auth Check: IsAuthenticated ✓
> Rate Limit: 5 audits/hr max ✓
> State Check: No active audits RUNNING ✓

Result: 202 Accepted. New Audit UUID generated.

02. Background Execution Asynchronous

The task is handed off to a Celery worker using Redis as the message broker. This ensures your dashboard stays fast while we do the heavy lifting.

  • Fetches encrypted AWS/GitHub credentials from Vault.
  • Decrypts tokens in-memory only.
  • Queries external APIs for real-time configurations.

03. Evidence Evaluation

Every resource is checked against the compliance ruleset. We handle exceptions through a Risk Acceptance logic.

if check.status == 'FAIL':
  if RiskAcceptance.exists(resource_id):
    status = 'RISK_ACCEPTED' # Override
  else:
    status = 'FAIL'

04. Snapshot & Integrity

Once completion is reached, a point-in-time snapshot is frozen. This data is now immutable.

  • SHA-256 Checksum: Generated for the entire audit result to prove no tampering.
  • Score Calculation: (Pass + Risk Accepted) / Total Checks.

05. Redacted Export

When sharing reports with external auditors, AuditMate applies a Redaction Layer.

Sensitive data like Internal IP addresses, DB secrets, and private repo names are replaced with generic placeholders while maintaining the PASS/FAIL integrity.

Failure Modes & Recovery

We use exponential backoff for API 429 errors. If a worker crashes, a cleanup cron job identifies the stale "RUNNING" audit and marks it as FAILED within 60 minutes to maintain system accuracy.
